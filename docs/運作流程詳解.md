# Copilot Ralph Node 運作流程詳解

本文件以 **copilot-ralph-node** 為範圍，完整說明從 CLI 啟動、組裝提示、建立 SDK 連線、進入迴圈、事件串流輸出、到結束與清理的全流程。所有程式碼參考都附上檔名/路徑連結，方便對照實作。

## 目的與範圍

- 目的：釐清 Ralph Node 版本的實際運作路徑與每個階段的行為細節。
- 範圍：僅涵蓋本專案 `copilot-ralph-node` 的程式碼與 CLI 行為。
- 版本依據：以目前 `src/` 內的 TypeScript 實作為主。

## 整體架構與資料流

核心流程可概括為：

1. CLI 入口解析參數並建立 `LoopConfig`。
2. 解析 prompt（字串、Markdown 或文字檔案）。
3. 建立 SDK Client 並啟動。
4. `LoopEngine.start()` 開始迴圈。
5. 透過 SDK 事件流取得 AI 回覆與工具事件。
6. 事件持續輸出到 CLI。
7. 完成/失敗/取消後產生摘要並回傳退出碼。

簡化資料流示意：

```
CLI 命令列
  └─ src/cli-entry.ts
       └─ src/cli/commands/run-command.ts
            ├─ 建立 LoopConfig
            ├─ createSDKClient()
            ├─ LoopEngine.start()
            │    ├─ 發送 prompt
            │    ├─ 事件流（AI 回覆 / 工具呼叫）
            │    └─ promise 偵測與完成
            └─ 顯示事件與摘要輸出
```

## CLI 啟動入口與命令處理

### 入口檔案與命令註冊

- CLI 入口為 [`src/cli-entry.ts`](../src/cli-entry.ts)。
  - 使用 `commander` 設定指令與全域參數。
  - `run` 指令由 `createRunCommand()` 註冊。

### `run` 指令流程

`run` 指令實作在 [`src/cli/commands/run-command.ts`](../src/cli/commands/run-command.ts)，核心流程如下：

1. **解析與套用參數**：`createRunCommand()` 透過 Commander 解析 CLI 參數。
2. **resolvePrompt**：`resolvePrompt()` 會檢查參數是否為 `.md`/`.markdown`/`.txt` 檔案路徑；是的話讀檔內容，不是則視為原始 prompt 字串。
3. **buildLoopConfig**：組裝 `LoopConfig`（含 `promisePhrase`、`maxIterations` 等）。
4. **validateRunConfig / validateSettings**：檢查 `prompt`、`timeout`、`maxIterations`、`systemPromptMode` 等合法性。
5. **dry-run 分支**：`--dry-run` 只輸出設定，不進入迴圈。
6. **建立 SDK Client**：`createSDKClient()` 將設定轉換為 SDK 選項並回傳 client。
7. **啟動 SDK**：CLI 層先呼叫 `sdkClient.start()`。
8. **進入 LoopEngine**：建立 `LoopEngine`，呼叫 `engine.start()` 進入迴圈。
9. **顯示事件與摘要**：`displayEvents()` 監聽事件流，結束後輸出 `printSummary()` 與退出碼。

## 參數與設定來源

`run` 指令的參數與預設值主要在 [`src/cli/commands/run-command.ts`](../src/cli/commands/run-command.ts) 與 [`src/core/loop-config.ts`](../src/core/loop-config.ts) 內定義。

常用參數：

- `--max-iterations`：最大迭代次數（預設 10）。
- `--timeout`：最大執行時間（預設 30 分）。
- `--promise`：完成短語（預設 `任務完成！🥇`）。
- `--model`：模型名稱（預設 `gpt-5-mini`）。
- `--working-dir`：工作目錄（預設 `.`）。
- `--streaming`：是否開啟串流（預設 `true`）。
- `--system-prompt` / `--system-prompt-mode`：自訂系統訊息或 append/replace 模式。
- `--system-prompt-file`：系統提示模板檔案路徑（支援 `{{PROMISE}}` 取代）。
- `--log-level`：SDK 日誌層級（debug/info/warn/error）。
- `--dry-run`：僅輸出設定不執行。

### LoopConfig 組成

`LoopConfig` 型別在 [`src/core/loop-config.ts`](../src/core/loop-config.ts) 定義，包含：

- `prompt`
- `promisePhrase`
- `model`
- `workingDir`
- `maxIterations`
- `timeoutMs`
- `dryRun`

## Prompt 與 System Prompt 的組裝

### 使用者 prompt 解析

`resolvePrompt()` 位於 [`src/cli/commands/run-command.ts`](../src/cli/commands/run-command.ts)。行為：

- 若輸入不是檔案路徑 → 原樣使用。
- 若是 `.md`/`.markdown`/`.txt` → 讀取檔案文字。
- 若是目錄或副檔名非 `.md`/`.markdown`/`.txt` → 直接拋錯。

### System Prompt 組裝

System Prompt 的模板與插值流程：

- 預設模板內容已內嵌於 [`src/core/system-prompt.ts`](../src/core/system-prompt.ts)。
  - 參考文本仍保留在 [`src/core/system.md`](../src/core/system.md)。
  - 內含 `<promise>{{PROMISE}}</promise>` 完成訊號規則。
- 取代：[`src/core/system-prompt.ts`](../src/core/system-prompt.ts) 的 `buildSystemPrompt()`。

若有自訂 system prompt 或模板：

- `--system-prompt` 由 [`src/cli/commands/run-command.ts`](../src/cli/commands/run-command.ts) 解析（文字或檔案）；提供時會忽略 `--system-prompt-file` 與內建模板。
- `--system-prompt-file` 可指定模板檔案路徑（支援 `{{PROMISE}}` 取代）。
- `--system-prompt-mode` 為 `append` 或 `replace`，會傳入 SDK 的 system message mode（搭配 `--system-prompt` 或 `--system-prompt-file`）。

## SDK Client 的建立與啟動

SDK 建立流程在 [`src/cli/commands/run-command.ts`](../src/cli/commands/run-command.ts) 的 `createSDKClient()`：

- 透過 `withModel`、`withWorkingDir`、`withTimeout`、`withStreaming`、`withLogLevel` 將 CLI 設定轉成 SDK 參數。
- `withSystemMessage` 接收 System Prompt（預設模板或自訂內容）。
- SDK 實際封裝在 [`src/sdk/client.ts`](../src/sdk/client.ts)。

注意：目前流程會在 CLI 層先呼叫一次 `sdkClient.start()`，接著在 `LoopEngine.start()` 內再次呼叫 `sdk.start()`（見 [`src/core/loop-engine.ts`](../src/core/loop-engine.ts)）。這是現行行為的一部分。

## LoopEngine 的生命週期與迴圈細節

LoopEngine 核心在 [`src/core/loop-engine.ts`](../src/core/loop-engine.ts)，主要流程如下：

### 1) `start()` 初始化

- 建立新的 `AbortController`，以 CLI signal 對接。
- 設定 timeout（到期會 `abort(ErrLoopTimeout)`）。
- 初始化 state、iteration、promiseDetected。
- 發送 `LoopStartEvent`（[`src/core/loop-events.ts`](../src/core/loop-events.ts)）。
- 若有 SDK：
  - `sdk.start()`
  - `sdk.createSession()`
- 進入 `runLoop()`。

### 2) `runLoop()` 迴圈

每次迴圈會：

1. `preIterationCheck()`：
   - 檢查 `signal.aborted` 或 `state === cancelled`。
   - 檢查 timeout。
   - 若 `iteration >= maxIterations` → `complete()`。
2. `executeIteration()`：
   - `iteration += 1`。
   - 發送 `IterationStartEvent`。
   - 組合 prompt：`[Iteration N/max]\n\n{prompt}`。
   - `sdk.sendPrompt()` 取得事件流。
3. 執行完該 iteration 後，若 `promiseDetected === true` → `complete()`。

### 3) `executeIteration()` 與事件處理

事件類型與對應行為：

- `TextEvent`（[`src/sdk/events.ts`](../src/sdk/events.ts)）：
  - 發送 `AIResponseEvent`（[`src/core/loop-events.ts`](../src/core/loop-events.ts)）。
  - 若 `reasoning === false` 且偵測到 promise → 發送 `PromiseDetectedEvent` 並設 `promiseDetected = true`。
- `ToolCallEvent`：
  - 發送 `ToolExecutionStartEvent`。
- `ToolResultEvent`：
  - 發送 `ToolExecutionEvent`（含結果或錯誤）。
- `SdkErrorEvent`：
  - 發送 `ErrorEvent`，視為可復原錯誤訊號。

事件處理完成後，會發送 `IterationCompleteEvent`。

## Promise 偵測與完成條件

Promise 偵測邏輯在 [`src/core/promise-detection.ts`](../src/core/promise-detection.ts)：

- 完成短語為 `<promise>${promisePhrase}</promise>`。
- 比對採用 `text.includes()`，**區分大小寫且需完全一致**。
- 只有 **非 reasoning** 的 `TextEvent` 會進行偵測。

完成條件：

1. **偵測到 promise**：在該 iteration 結束後完成（`promiseDetected` 觸發）。
2. **達到 maxIterations**：`preIterationCheck()` 直接 `complete()`。
3. **超時**：`ErrLoopTimeout` → `failed`。
4. **取消**：`ErrLoopCancelled` → `cancelled`。

## 事件串流與 CLI 輸出

事件輸出邏輯位於 [`src/cli/commands/run-command.ts`](../src/cli/commands/run-command.ts) 的 `displayEvents()`：

- `LoopStartEvent` → 顯示「Loop started」。
- `IterationStartEvent` → 顯示「Iteration N/M」標題。
- `AIResponseEvent` → 直接寫到 stdout。
- `ToolExecutionStartEvent` / `ToolExecutionEvent` → 顯示工具呼叫與結果。
- `PromiseDetectedEvent` → 顯示「Promise detected」。
- `LoopCompleteEvent` / `LoopFailedEvent` / `LoopCancelledEvent` → 結束事件流並返回。

事件本身型別定義在 [`src/core/loop-events.ts`](../src/core/loop-events.ts)。

## 結束狀態與退出碼

退出碼對應關係在 [`src/cli/commands/run-command.ts`](../src/cli/commands/run-command.ts)：

- `exitSuccess = 0`：完成（`complete`）。
- `exitFailed = 1`：失敗。
- `exitCancelled = 2`：取消。
- `exitTimeout = 3`：超時。
- `exitMaxIterations = 4`：達到最大迭代次數（目前實作多由 `complete` 路徑結束，仍保留此碼）。

最終摘要輸出由 `printSummary()` 完成，同樣在 [`src/cli/commands/run-command.ts`](../src/cli/commands/run-command.ts)。

## 事件通道與 AsyncQueue 行為

事件通道實作在 [`src/shared/async-queue.ts`](../src/shared/async-queue.ts)：

- 支援 `AsyncIterable`，讓 `displayEvents()` 以 `for await` 讀取事件。
- 具 buffer size 限制（`LoopEngine` 使用 100）。
- `close()` 後會釋放等待的 consumer，避免掛起。

## 建置與執行模式

建置與執行流程參考 [`package.json`](../package.json)：

- `npm run dev`：使用 `tsx` 直接跑 `src/cli-entry.ts`。
- `bun run build`：輸出到 `dist/`，實際發行執行入口為 `dist/cli-entry.js`（參見 `bin` 設定）。

這代表如果你從全域安裝執行 `copilot-ralph`，多半是執行 `dist/cli-entry.js`。
